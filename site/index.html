What changes

Less scrolling. Better evidence.

A public request is useful demand evidence, but only after it has been checked for safety, feasibility, and relevance beyond one person's exact situation. Wishwright makes those checks explicit and keeps the source attached to every approved brief.

01

Rank the batch

The fixture source reads one post per JSONL line. The evaluator assigns safety, feasibility, breadth, and total scores from 0 to 1, then the CLI prints candidates from strongest to weakest.

02

Stop unsafe requests

A case-insensitive deny-list is a hard gate. A matched term forces the total to zero, so a dangerous request cannot pass because it happens to sound feasible or widely useful.

03

Keep the trail

Evaluation events go to a timestamped JSONL run log. A locked ledger stores each candidate's pipeline stage and refreshes before writes, preventing stale processes from erasing newer records.

04

Hand off a brief

Approved candidates convert to a dictionary with the request text, source URL, public author, rationale, and component scores. That stable boundary lets a build system decide what happens next.

Run it locally

Inspect the whole decision.

Install the package on Python 3.11 or newer, score the included fixture, and inspect the JSONL audit log. Configuration is plain YAML, and the test path needs no API key or hosted database.

Current integration boundary X discovery, downstream build invocation, repository publishing, and reply posting are not wired into the CLI yet. The repository provides tested interfaces and local handoff objects for those stages.
Terminal / three commands
git clone https://github.com/ctkrug/build-a-public-request-to-product-system.git

cd build-a-public-request-to-product-system
pip install -e ".[dev]"

Questions

Know the boundary before you build.

Wishwright is a tested local pipeline core, not a hosted bot. These answers spell out what runs today and what an adopter still needs to connect.

What does Wishwright do today?

It reads local request fixtures, applies a configurable safety and usefulness rubric, prints a ranked shortlist, writes audit events, tracks pipeline stages, checks built repositories, drafts replies, and produces build-brief dictionaries.

How does safety screening work?

Policy terms are matched case-insensitively against request text. A match returns zero for every score and records the reason, while non-matches continue through feasibility and breadth scoring.

Does Wishwright connect to X?

Not in the current release. `XApiSource` marks the intended adapter boundary and raises until credentials and request handling are implemented. The included source uses deterministic JSONL fixtures.

What reaches the build system?

Only approved candidates can become briefs. Each brief includes the title, source candidate ID, source URL, public-request rationale, and all four evaluation scores.
